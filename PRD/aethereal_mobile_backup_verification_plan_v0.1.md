## Verification Plan

**Version:** 0.1  
**Aligned PRD:** Aethereal Mobile Backup Appliance PRD v0.2  
**Aligned implementation:** Implementation Plan v0.1

---

# 1. Purpose

This document defines how the Aethereal Mobile Backup Appliance shall be verified before version 1 is considered complete.

The verification strategy focuses on:

- Source-media protection.
- Backup correctness.
- Capacity prediction.
- File identity.
- Cryptographic verification.
- Interrupted-operation recovery.
- Physical media removal.
- Web and LED state consistency.
- Field usability.

The verification goal is not only to prove that files copy successfully under normal conditions.

The system must also prove that abnormal conditions do not cause incomplete or unverified files to be reported as successful backups.

---

# 2. Verification Principles

The following principles apply to all verification activities.

## VP-001 - Integrity Over Throughput

A test failure affecting source protection, file verification, recovery, or backup completion classification shall be considered more severe than a performance failure.

## VP-002 - Filesystem Inspection

Tests shall inspect the destination filesystem directly.

Passing UI status alone is not sufficient.

## VP-003 - Database Inspection

Tests shall inspect SQLite state where relevant.

Passing filesystem state alone is not sufficient.

## VP-004 - Failure Injection

Failure scenarios shall be intentionally injected.

Unexpected power loss, process termination, source removal, destination removal, and verification failure shall not be simulated only through graceful API paths.

## VP-005 - Requirement Traceability

Every PRD requirement shall map to at least one:

- Automated test,
- System test,
- Hardware qualification test,
- Manual acceptance test.

---

# 3. Test Levels

Verification shall use five test levels.

## 3.1 Unit Tests

Purpose:

- Validate isolated logic.

Examples:

- State-machine transitions.
- Safety-margin calculations.
- Capacity calculations.
- Path sanitization.
- Content-identity comparison.
- LED progress bucket calculation.
- Retry-count logic.

Target:

- Run on development workstation and Raspberry Pi.

---

## 3.2 Integration Tests

Purpose:

- Validate cooperation between application modules.

Examples:

- SQLite state plus backup planner.
- Inventory plus content identity.
- Copy plus verification.
- Event publication plus web status.
- Device identity plus destination selection.

Use temporary filesystems and loopback devices where possible.

---

## 3.3 Fault-Injection Tests

Purpose:

- Validate recovery and integrity under abnormal conditions.

Examples:

- Kill `backupd`.
- Remove source media.
- Remove destination SSD.
- Force hash mismatch.
- Fill destination filesystem.
- Corrupt SQLite transaction state.
- Power-cycle the Raspberry Pi.

---

## 3.4 System Tests

Purpose:

- Validate the complete appliance.

Tests shall use:

- Raspberry Pi 4.
- Qualified power source.
- Qualified SSD.
- Qualified SD reader.
- Wi-Fi access point.
- iPhone client.
- Physical LED.

---

## 3.5 Field Acceptance Tests

Purpose:

- Validate realistic photographer workflow.

Tests shall use real media generated by:

- Canon EOS R6 Mark II.
- DJI Mini 5 Pro.

The field workflow shall be completed without shell access.

# 6. Automated Unit Verification

## UT-001 - State Transition Validation

Verify every allowed state transition.

Verify representative invalid transitions are rejected.

Examples:

```text
IDLE -> BACKUP_COMPLETED = reject
PREFLIGHT_BLOCKED -> BACKUP_COPYING = reject
BACKUP_COPYING -> BACKUP_CANCELLING = allow
```

Pass criteria:

- Invalid transitions raise a controlled error.
- State is not changed.
- Error is logged.

PRD coverage:

- Section 20
- DB-004
- WEB-007 architecture principle

---

## UT-002 - Safety Margin Calculation

Test:

```text

small destination
```

# 26. Defect Severity

## Critical

Examples:

- Source media modified.
- Unverified file reported as verified.
- Partial file reported as successful backup.
- Wrong destination device used.
- Backup reported complete with missing required files.
- Data overwritten without conflict detection.

Release rule:

```text
0 open critical defects
```

---

## High

Examples:

- Interrupted job cannot recover.
- Insufficient-capacity backup starts.
- Hash mismatch is ignored.
- Source or destination removal is not detected.
- Backup stops when iPhone disconnects.

Release rule:

```text
0 open high-severity integrity defects
```

---

## Medium

Examples:

- Incorrect ETA.
- Delayed dashboard refresh.
- Non-critical LED timing deviation.
- Log filter malfunction.

Release rule:

- May be accepted with documented workaround and explicit release decision.

---

## Low

Examples:

- Minor UI layout issue.
- Cosmetic text problem.
- Non-blocking log formatting problem.

---

# 27. Release Exit Criteria

Version 1 may be released only when:

1. All critical acceptance tests pass.
2. All PRD v1 acceptance criteria are traced to passing verification evidence.
3. No critical defects remain open.
4. No high-severity integrity defects remain open.
5. Source read-only protection has passed on the reference hardware.
6. Power-loss recovery tests have passed.
7. Destination-removal and source-removal tests have passed.
8. Canon field acceptance has passed.
9. DJI field acceptance has passed.
10. The 24-hour soak test has passed.
11. Reference hardware configuration is documented.
12. Installation from a fresh supported Raspberry Pi OS image has been verified.
13. The final application version, database schema version, OS version, and hardware configuration are recorded in the release evidence.

# 29. Final Verification Principle

A successful test is not:

> The files appeared on the SSD.

A successful verification demonstrates that:

- The source was protected.
- The destination was correct.
- Capacity was sufficient.
- The planned files were processed.
- Copied content matched the source.
- Incomplete content was never presented as valid.
- The system recovered correctly when interrupted.
- The user received an accurate final state.
